01_Connection

Core Requires PBI Desktop

Detect running Power BI Desktop instances and establish a live connection for all subsequent operations.

Operations

detectScan for running PBI instances
connectConnect to a specific instance

Parameters

ParameterTypeDefaultDescription
operation"detect" | "connect"detectOperation to perform
model_indexinteger0Index of the model to connect to (connect only)
Usage Examples
// Detect all running instances {"operation": "detect"} // Connect to first instance {"operation": "connect", "model_index": 0} // Connect to second instance {"operation": "connect", "model_index": 1}
What You Can Do
  • Find all open PBI files — run detect to see every running instance with port, PID, and file name
  • Switch between instances — call connect with different model_index values to switch
  • Multi-instance debugging — connect to one instance, run queries, then switch to another for comparison
  • Auto-detect PBIP path — on connect, the server auto-detects the PBIP folder if the file is from a PBIP project

02_Model_Operations

Model Live Connection

Unified CRUD for all model objects: tables, columns, measures, relationships, and calculation groups. The workhorse tool for model inspection and modification.

Object Types & Operations

Object TypeAvailable Operations
tablelist, describe, sample_data, create, update, delete, refresh, generate_calendar
columnlist, get, statistics, distribution, create, update, delete
measurelist, get, create, update, delete, rename, move
relationshiplist, get, find, create, update, delete, activate, deactivate
calculation_grouplist, create, delete, list_items
What You Can Do
  • Explore model structure — list tables, describe columns, inspect relationships
  • Preview data — sample_data with column selection, ordering, and row limits
  • Create measures — add new DAX measures with format strings and display folders
  • Manage relationships — create, activate/deactivate, set cross-filtering behavior
  • Generate calendar tables — auto-generate date tables with fiscal year support
  • Column statistics — get min/max/distinct/null counts and value distributions
  • Organize measures — rename, move between tables, set display folders
Key Parameters
ParameterTypeDescription
object_type REQUIREDenumtable | column | measure | relationship | calculation_group
operationstringOperation (varies by object_type)
table_namestringTable name
expressionstringDAX expression
format_stringstringFormat string (e.g., #,##0.00)
display_folderstringDisplay folder path
max_rowsintegerMax rows for sample_data (default: 10, max: 1000)
hiddenbooleanHide object from client tools
Usage Examples
// List all tables {"object_type": "table", "operation": "list"} // Preview Sales data {"object_type": "table", "operation": "sample_data", "table_name": "Sales", "max_rows": 20} // Create a measure {"object_type": "measure", "operation": "create", "table_name": "Sales", "measure_name": "Total Sales", "expression": "SUM(Sales[Amount])", "format_string": "#,##0.00"} // Create a relationship {"object_type": "relationship", "operation": "create", "from_table": "Sales", "from_column": "ProductKey", "to_table": "Products", "to_column": "ProductKey"}

03_Batch_Operations

Batch Live Connection

Execute batch operations with ACID transaction support. 3-5x faster than individual calls. All-or-nothing with rollback on failure.

Object Types & Batch Operations

  • Atomic updates — all succeed or all roll back with use_transaction: true
  • Dry run validation — validate all definitions without executing
  • Error resilience — optionally continue on errors with continue_on_error: true

04_Run_DAX

Query Live Connection

Execute DAX queries with automatic row limits and optional SE/FE trace analysis for performance debugging.

Execution Modes

ModeDescriptionUse Case
auto / simpleExecute and return preview resultsQuick data inspection
analyzeMulti-run benchmark with timing statisticsPerformance comparison
traceSE/FE timing analysis with query eventsDeep performance debugging
What You Can Do
  • Run any DAX query — EVALUATE statements with automatic top-N limiting
  • Performance trace — get SE/FE timing, cache hit ratios, SE query count
  • Benchmark queries — multi-run analysis in analyze mode
  • Cold cache testing — clear VertiPaq cache before trace for realistic timing
Usage Examples
// Simple query {"query": "EVALUATE SUMMARIZECOLUMNS('Date'[Year], \"Total\", [Total Sales])"} // With SE/FE trace {"query": "EVALUATE SUMMARIZECOLUMNS('Date'[Year], \"Total\", [Total Sales])", "mode": "trace", "clear_cache": true}
